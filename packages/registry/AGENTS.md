# Registry Package

Git-based module and template registry for 86d. Resolves, fetches, and caches modules/templates from local workspace, GitHub repositories, or npm packages.

## Structure

```
src/
  index.ts          Public API re-exports
  types.ts          Zod schemas + TypeScript types (manifest, specifier, config)
  specifier.ts      Parse module specifier strings into structured objects
  resolver.ts       Resolve config → concrete module list (local, registry, GitHub, npm)
  fetcher.ts        Download modules from GitHub tarballs or npm (with retry + integrity)
  manifest.ts       Build registry.json from local modules/ and templates/ directories
  lockfile.ts       Lock file (registry.lock.json) generation, reading, and verification
  template.ts       Resolve and fetch templates (local, registry, GitHub, npm)
  config.ts         Read and normalize store config.json
  __tests__/        101+ tests covering all modules
```

## Module Specifier Formats

```
"products"                            → official (local or registry)
"@86d-app/products"                   → official (local or registry)
"github:owner/repo/modules/custom"    → GitHub subpath
"github:owner/repo/path#v2.0"         → GitHub with ref
"npm:@acme/module"                    → npm package
"npm:@acme/module@^1.0.0"             → npm with version
```

## Config Format

```json
{
  "modules": "*",
  "template": "brisa",
  "registry": "https://...registry.json",
  "moduleOptions": { "@86d-app/cart": { "maxItemsPerCart": 50 } }
}
```

- `modules: "*"` or omitted → all local workspace modules + registry modules
- `modules: [...]` → explicit list with missing module warnings
- `template` → local name, `github:owner/repo/templates/custom`, or `npm:@acme/theme`

## Key Functions

- `parseSpecifier(raw)` → `ModuleSpecifier` (source, name, packageName, repo, path, ref)
- `resolveModules(config, { root, manifest? })` → `ResolvedModule[]`
- `fetchModule(spec, root, manifest?)` → `FetchResult` (downloads from GitHub/npm)
- `getModuleDependencies(name, manifest)` → `string[]` (transitive deps, topologically sorted; throws on cycles)
- `detectCircularDependencies(manifest)` → `string[]` (cycle descriptions, empty if acyclic)
- `generateLockfile(resolved, root)` → `Lockfile` (snapshot resolved state for reproducible builds)
- `writeLockfile(root, lockfile)` / `readLockfile(root)` — persist/load `registry.lock.json`
- `verifyLockfile(lockfile, resolved)` → `LockfileDiff` (added/removed/changed modules)
- `isLockfileSatisfied(diff)` → `boolean` (true if no changes)
- `resolveTemplate(spec, root, manifest?)` → `ResolvedTemplate`
- `fetchTemplate(spec, root, manifest?)` → `FetchResult`
- `buildManifest(root)` → `RegistryManifest` (scans modules/ + templates/)
- `readStoreConfig(path)` → `StoreConfig`
- `getLocalModuleNames(root)` → `string[]`
- `getLocalTemplateNames(root)` → `string[]`
- `computeIntegrity(modulePath)` → `string | undefined`

## Registry Manifest (`registry.json`)

Generated by `scripts/generate-registry.ts`. Indexes all modules and templates:
- modules: name, description, version, category, path, requires, integrity (sha256)
- modules: hasStoreComponents, hasAdminComponents, hasStorePages
- templates: name, description, version, path

## Integration Points

- **`scripts/generate-modules.ts`** — uses `resolveModules()` + `fetchModule()` at buildtime to resolve all modules from config, fetch missing ones, then generate the store's import files. Generates `registry.lock.json` for reproducible builds. Supports `--frozen` flag for CI (fails if lock file is outdated). Runs `detectCircularDependencies()` before generating.
- **`packages/cli/src/commands/module.ts`** — uses `parseSpecifier()`, `fetchModule()`, `getModuleDependencies()`, `computeIntegrity()`, `getLocalModuleNames()`, `readLocalManifest()` for `86d module add/update/list/search/info`
- **`scripts/generate-registry.ts`** — uses `buildManifest()` to generate `registry.json`
- **`apps/store/next.config.ts`** — reads generated `transpile-packages.json` for Next.js config

## Lock File (`registry.lock.json`)

Generated by `scripts/generate-modules.ts`, committed to the repo. Captures the exact resolved state (source, version, integrity hash, path) of every module for reproducible builds. Use `--frozen` in CI to fail if the lock file doesn't match the current resolution.

## Gotchas

- `exactOptionalPropertyTypes` is enabled — use conditional assignment for optional props
- GitHub fetching uses tarball API with retry + exponential backoff; set `GITHUB_TOKEN` for private repos or rate limits
- npm modules install to `node_modules/`, not `modules/`; templates install to `templates/`
- The resolver checks local workspace first before falling back to registry
- Integrity verification: fetched modules are verified against sha256 hashes in the manifest
- Fetcher retries on HTTP 408, 429, 500, 502, 503, 504 and network errors
- `getModuleDependencies()` now throws on circular dependencies instead of silently skipping
