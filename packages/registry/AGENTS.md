# Registry Package

Git-based module and template registry for 86d. Resolves, fetches, and caches modules/templates from local workspace, GitHub repositories, or npm packages.

## Structure

```
src/
  index.ts          Public API re-exports
  types.ts          Zod schemas + TypeScript types (manifest, specifier, config)
  specifier.ts      Parse module specifier strings into structured objects
  resolver.ts       Resolve config ‚Üí concrete module list (local, registry, GitHub, npm)
  fetcher.ts        Download modules from GitHub tarballs or npm (with retry + integrity)
  manifest.ts       Build registry.json from local modules/ and templates/ directories
  template.ts       Resolve and fetch templates (local, registry, GitHub, npm)
  config.ts         Read and normalize store config.json
  __tests__/        87+ tests covering all modules
```

## Module Specifier Formats

```
"products"                            ‚Üí official (local or registry)
"@86d-app/products"                   ‚Üí official (local or registry)
"github:owner/repo/modules/custom"    ‚Üí GitHub subpath
"github:owner/repo/path#v2.0"         ‚Üí GitHub with ref
"npm:@acme/module"                    ‚Üí npm package
"npm:@acme/module@^1.0.0"             ‚Üí npm with version
```

## Config Format

```json
{
  "modules": "*",
  "template": "brisa",
  "registry": "https://...registry.json",
  "moduleOptions": { "@86d-app/cart": { "maxItemsPerCart": 50 } }
}
```

- `modules: "*"` or omitted ‚Üí all local workspace modules + registry modules
- `modules: [...]` ‚Üí explicit list with missing module warnings
- `template` ‚Üí local name, `github:owner/repo/templates/custom`, or `npm:@acme/theme`

## Key Functions

- `parseSpecifier(raw)` ‚Üí `ModuleSpecifier` (source, name, packageName, repo, path, ref)
- `resolveModules(config, { root, manifest? })` ‚Üí `ResolvedModule[]`
- `fetchModule(spec, root, manifest?)` ‚Üí `FetchResult` (downloads from GitHub/npm)
- `getModuleDependencies(name, manifest)` ‚Üí `string[]` (transitive deps, topologically sorted)
- `resolveTemplate(spec, root, manifest?)` ‚Üí `ResolvedTemplate`
- `fetchTemplate(spec, root, manifest?)` ‚Üí `FetchResult`
- `buildManifest(root)` ‚Üí `RegistryManifest` (scans modules/ + templates/)
- `readStoreConfig(path)` ‚Üí `StoreConfig`
- `getLocalModuleNames(root)` ‚Üí `string[]`
- `getLocalTemplateNames(root)` ‚Üí `string[]`
- `computeIntegrity(modulePath)` ‚Üí `string | undefined`

## Registry Manifest (`registry.json`)

Generated by `scripts/generate-registry.ts`. Indexes all modules and templates:
- modules: name, description, version, category, path, requires, integrity (sha256)
- modules: hasStoreComponents, hasAdminComponents, hasStorePages
- templates: name, description, version, path

## Integration Points

- **`scripts/generate-modules.ts`** ‚Äî uses `resolveModules()` + `fetchModule()` at buildtime to resolve all modules from config, fetch missing ones, then generate the store's import files
- **`packages/cli/src/commands/module.ts`** ‚Äî uses `parseSpecifier()`, `fetchModule()`, `getModuleDependencies()`, `computeIntegrity()`, `getLocalModuleNames()`, `readLocalManifest()` for `86d module add/update/list/search/info`
- **`scripts/generate-registry.ts`** ‚Äî uses `buildManifest()` to generate `registry.json`
- **`apps/store/next.config.ts`** ‚Äî reads generated `transpile-packages.json` for Next.js config

## Gotchas

- `exactOptionalPropertyTypes` is enabled ‚Äî use conditional assignment for optional props
- GitHub fetching uses tarball API with retry + exponential backoff; set `GITHUB_TOKEN` for private repos or rate limits
- npm modules install to `node_modules/`, not `modules/`; templates install to `templates/`
- The resolver checks local workspace first before falling back to registry
- Integrity verification: fetched modules are verified against sha256 hashes in the manifest
- Fetcher retries on HTTP 408, 429, 500, 502, 503, 504 and network errors
